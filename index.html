PROBLEM · 痛点发现

工具换了一个又一个，
知识还是一盘散沙

痛点 1

只收不用

老师的课堂内容、大家分享的优质知识收了一大堆，却从来不回头看。收藏的那一刻就是被遗忘的开始。

痛点 2

信息孤岛

笔记各自为政，想法之间没有连接。知识越积越多，越来越找不到。

痛点 3

维护负担

分类、打标签、定期整理……维护笔记本身就耗尽精力，知识库最终烂尾。

核心问题：知识管理的维护成本随信息量线性增长——这是人力无法逾越的天花板。
不是工具不好，是"人来维护知识"这件事本身就有天花板。

COMPETITIVE RESEARCH · 竞品研究

市面上的解法，
各有致命短板

维度 飞书文档 / Notion Google NotebookLM 我的需求
AI 整理 ✗ 无，全靠人工 ✓ 自动摘要、问答 ✓ 必须有
跨库串联 ✗ 知识碎片化 ✗ 知识库彼此孤立 ✓ 核心需求
容量限制 — 有上限 ✗ 每个 Notebook 有上限 ✓ 无限制
本地化 — 云端 ✗ 纯云端，数据不自主 ✓ 本地优先
知识自更新 ✗ 静态文档 ✗ 上传即固定 ✓ 增量编译
共同盲点：知识库是孤立的容器——不同领域的知识无法真正串联，而真实的思维从来不是这样工作的。
NotebookLM 最接近，但跨库孤立和容量限制是根本性缺陷。

KEY INSIGHT · 2026-04-02

Karpathy 的一条推文，
让我看到了第三条路

@karpathy · April 2, 2026

"Using LLMs to build personal knowledge bases… a large fraction of my recent token throughput is going less into manipulating code, and more into manipulating knowledge."

洞见 1

知识库能自我更新

LLM 充当编译器，新素材自动合并进已有体系。知识库是活的，不是静态文档。

洞见 2

知识能互相连接

AI 维护跨主题双向链接，不同领域的知识自由串联，形成真正的知识图谱。

决定：结合本地 Obsidian，自建一套不依赖云端的个人知识图谱。

PRODUCT DEFINITION · 产品定义

从痛点和竞品中，
提炼出三条核心需求

01

跨主题串联，没有孤岛

解决飞书 / Notion 碎片化，以及 NotebookLM 知识库孤立的问题。

02

AI 维护，人只做方向决策

解决维护成本天花板，把所有重复性整理工作完全交给 AI。

03

本地化，数据自主，无上限

解决 NotebookLM 云端依赖和容量限制，结合 Obsidian 本地自建。

三条需求直接对应痛点和竞品分析，有逻辑推导，不是拍脑袋定的。

ITERATIONS · 迭代历程

MVP 之后，架构经历了四次进化

v1

基础骨架 — 线性流水线

建立核心目录，跑通「投入 → 编译 → 输出」基本流程。

v2

wiki/ 目录分级

平铺 → 「领域 / 项目」两级，不同项目物理隔离，知识不再混杂。

v3

引入 Audit + Log — 反馈闭环

新增人工反馈通道和操作日志，系统从单向输出升级为可追溯闭环。

v4

集成 SKILL 系统 — 输出扩展

引入可插拔 SKILL 插件机制，首个落地：一条指令生成 PPT。

每次迭代来自真实使用暴露的问题，不是提前规划的——系统跑起来才知道自己需要什么。

ROADMAP · my_wiki + Obsidian vs NotebookLM

对标 NotebookLM，
我们已超越 · 还在追

✓ 已实现

跨来源综合问答

NotebookLM 知识库彼此孤立，无法跨库问答。my_wiki 以「综合 [主题]」指令跨全部 wiki 页面生成分析，猎头 × 营销风控跨项目串联已落地。

计划中

Audio Overview 播客摘要

NotebookLM 最受欢迎的功能：将资料自动生成两人对话式音频。计划以 SKILL 插件形式集成，对任意 wiki 主题生成音频摘要。

计划中

结构化输出：学习指南 / FAQ / 时间线

NotebookLM 一键生成学习指南、常见问题、时间线等固定格式。计划扩展「导出」指令，以 SKILL 插件支持多种结构化模板。

✓ 已实现

Mind Map 知识图谱可视化

NotebookLM 的 Mind Map 为静态生成。Obsidian 图谱视图基于 wiki 双向链接实时渲染，节点随知识库增长自动扩展，可交互探索。

本地架构的优势：不受云端限制，功能可按需用 SKILL 插件扩展，数据完全自主。
